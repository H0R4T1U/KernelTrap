Features

Everything you need to
detect and contain threats

A complete intrusion detection pipeline from kernel-level collection to automated attacker containment.

⚡

eBPF Syscall Monitoring

Uses Tracee (eBPF) to collect every syscall event from the Linux kernel — zero overhead, no kernel patches, no application changes required.

🤖

ML Anomaly Detection

Isolation Forest model trained on the BETH honeypot dataset scores each event in real time. Detects behavioral anomalies, not just known signatures.

📷

Honeypot Pivot

After sustained high-severity activity, the SSH session is transparently redirected into an isolated Docker honeypot — without alerting the attacker.

📊

Real-time Dashboard

React operator dashboard with live event feed over WebSocket, per-user anomaly pressure gauges, pivot history, and manual pivot controls.

🔒

SSH-scoped Tracking

Only remote SSH sessions are tracked. Physical console users and background system daemons are excluded, eliminating false positives from local activity.

⚙

Multi-agent Support

Agents register with the central server over Redis Streams. Multiple monitored machines report to a single server with per-host isolation.

How it works

From syscall to honeypot
in milliseconds

KernelTrap operates in a continuous pipeline — collecting, scoring, and acting on kernel events as they happen.

01

Collect syscall events via eBPF

Tracee runs as a Docker container with privileged access to the Linux kernel. It attaches eBPF probes and emits a JSON stream of syscall events including process ID, user ID, syscall name, arguments, and return values in real time.

02

Filter to active SSH sessions

syscall_logger.py (the agent) checks which UIDs have an active pts/ session via who, refreshing every 30 seconds. Only events from remote SSH users are forwarded — local users and system daemons are silently dropped.

03

Stream to central server via Redis

The agent batches events and publishes them to events.{hostname} Redis Streams every 5 seconds. The central server subscribes to all agent streams simultaneously and picks up new agents instantly via a registration channel.

04

Score with Isolation Forest

The central server scores each event with the trained Isolation Forest model (BETH dataset, 7-feature vector). Events are classified as severity 0 (normal), 1 (low anomaly), or 2 (high anomaly). Scores are published to scores.{hostname} and broadcast to the dashboard over WebSocket.

05

Track anomaly window per user

A 60-second sliding window counts severity-2 events per (hostname, userId) pair. Once the threshold is crossed, a pivot command is published to commands.{hostname}. The operator can also trigger a manual pivot from the dashboard at any time.

06

Pivot attacker to honeypot

The agent's command listener reads the pivot command and executes hp_pivot_user, which drops the attacker's traffic via iptables and sends SIGUSR1 to their shell — transparently exec'ing into an SSH session pointing to the Docker honeypot on localhost:2222.
